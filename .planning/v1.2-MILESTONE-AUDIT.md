## Cross-Phase Integration

### Verified Flows

- `data/links.json` eligible social/profile destinations flow through `src/lib/ui/profile-quick-links.ts` into `src/routes/index.tsx`, then into `src/components/profile/ProfileHeader.tsx` and `src/components/profile/ProfileQuickLinks.tsx` without a second registry.
- The visible Quick Links strip implemented in Phase 16 matches the Phase 15 derivation rules and the Phase 17 maintainer docs/verification guide.
- The maintainer docs now consistently describe Quick Links as derived behavior from `links[]`, not `profileLinks`, across `docs/data-model.md`, `docs/customization-catalog.md`, and `README.md`.
- `docs/social-card-verification.md` now points to the shipped Quick Links tests instead of implying a new or heavier verification subsystem.

### Critical Gaps

None.

## End-to-End Flow Audit

### Visitor-facing flow

1. Visitor lands on the profile page.
2. If eligible social/profile links exist, the Quick Links strip appears above the action bar.
3. The strip uses icon-first outbound links with explicit accessible labels/titles and no selected state.
4. If no eligible links exist, the strip disappears and the existing action row remains intact.

Status: verified.

### Maintainer-facing flow

1. Maintainer updates existing top-level links in `data/links.json`.
2. Quick Links derive automatically; no second registry or profile-level shortcut list is needed.
3. Maintainer docs explain the behavior in `docs/data-model.md`, inventory it in `docs/customization-catalog.md`, and surface it briefly from `README.md`.
4. Maintainer verification guidance now explicitly covers the Quick Links strip and points to the focused automated tests.

Status: verified.
