# Phase 08.1: Custom Profile Descriptions - Context

**Gathered:** 2026-03-09
**Status:** Planned

## Phase Boundary

Separate profile-authored descriptions from fetched page/header descriptions for supported social profile links. This phase adds a new metadata surface and routes it through capture, rendering, and Studio authoring. Broader maintainer docs refresh and final milestone regression hardening remain in Phase 9.

## Implementation Decisions

### Data Contract

- Add `profileDescription` alongside the existing `description`; do not repurpose `description`.
- Keep `links[].description` and `descriptionSource` semantics focused on the generic page/card description path.
- Allow manual `metadata.profileDescription` to override generated or authenticated values field-by-field, matching the existing social profile override posture.
- Keep the new field optional and meaningful only for supported social profile links.

### Capture Audit

- Audit the currently supported social profile platforms and seeded links: Primal, GitHub, X, LinkedIn, Facebook, Instagram, YouTube, Medium, and Substack.
- Prefer public sources whenever they expose a distinct profile-authored description; only use authenticated paths where the repo already depends on authenticated cache support.
- If a platform exposes only one usable description surface, do not synthesize a duplicate `profileDescription`.
- Update repo-local extractor guidance so future platform work explicitly distinguishes page metadata from profile-authored descriptions when both exist.

### Rendering and Studio

- Social profile cards should prefer `profileDescription` when present, then fall back to the current description resolution rules.
- Non-profile links should keep current description behavior untouched.
- Studio should expose the new field in guided editing instead of forcing JSON-only edits.
- Any Studio implementation must update `docs/studio-phase-checklist.md` and `packages/studio-web/src/lib/phase-checklist.ts` in the same batch.

### Verification

- Add regression coverage for X plus at least one non-X supported public platform and one fallback or authenticated path.
- Validate with `bun run validate:data`, `bun run enrich:rich:strict`, targeted `bun run public:rich:sync`, `bun run build`, and the Studio lint/typecheck/integration suite if guided editor changes land.

## Specific Ideas

- X is the first audit target because the current oEmbed description is generic and known to diverge from the profile-authored bio.
- Use `profileDescription` rather than `bio` to match the existing `profileImage` naming pattern on link metadata.
- Keep extractor instructions explicit about capturing both page description and profile description when applicable, instead of asking contributors to choose one ad hoc.

## Deferred Ideas

- Additional platform-specific identity fields beyond `profileDescription`.
- Runtime toggles between profile-authored and generic page descriptions.
- Reworking non-profile card description semantics beyond the minimum fallback protections needed for this phase.
